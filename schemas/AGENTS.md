# Cross-service & cross-stack contracts

This is the ONLY allowed shared boundary between backend services and between
backend and frontend.

## Layout

- `openapi/<service>.json`     — REST API, AUTO-GENERATED by each backend service
- `proto/<service>/v<n>/*.proto`— gRPC contracts, hand-written, Buf-managed
- `events/*.cloudevents.json`  — async event schemas (CloudEvents 1.0)
- `codegen/`                   — generators (TS, Python, Go)

## Workflow

### Backend changes a REST endpoint
1. Edit `apps/backend/services/<svc>/src/.../routes/...`
2. From `apps/backend/`: `just gen-openapi <svc>`
3. → `schemas/openapi/<svc>.json` updated
4. From repo root: `just sync` regenerates ALL downstream clients

### Adding a gRPC method (internal)
1. Edit `schemas/proto/<svc>/v1/*.proto`
2. From `schemas/`: `buf lint && buf breaking`
3. From `apps/backend/`: `just gen-proto`
4. Implement server in `services/<svc>/src/.../grpc/`
5. Other services consume via `libs/transport/clients/<svc>/`

### Versioning
- REST: never break v1 silently; introduce v2 when needed
- gRPC: package versioning (`<svc>.v1`, `<svc>.v2`)
- Events: include version in filename (`bot.published.v1.json`)

## Rules for agents

- ⚠ NEVER hand-edit `openapi/*.json` (auto-regenerated)
- proto files are source of truth — edit them, not generated code
- `buf breaking` MUST pass; failures = breaking change → bump version
- Adding a new event: place in `events/`, document in `docs/微服务/events.md`

## What goes in `schemas/` vs `libs/`?

| Belongs in `schemas/`           | Belongs in `libs/`                  |
|---------------------------------|-------------------------------------|
| Wire formats (REST/gRPC/events) | Infrastructure code                 |
| Versioned contracts             | Shared transport clients            |
| Things consumers need to know   | Things consumers don't see          |
